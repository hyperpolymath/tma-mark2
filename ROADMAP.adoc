== Current Status

**Alpha scaffold (~25–30%).** A Phoenix shell exists. The Rust crypto/NLP NIFs work. The mark calculator works and is tested. Everything else listed in the README — CubDB-backed persistence, FHI XML parsing, file ingestion, `.docx` export, marking analytics, authentication — is either a docstring stub or partially-wired UI against mock data.

See link:TOPOLOGY.md[TOPOLOGY.md] for the component-by-component dashboard.

== Definition of v1

v1 is **a working replacement for the Java OU eTMA marking tool, for one tutor on one machine**. Nothing more.

=== In scope

* Watch a folder → ingest `.fhi` submissions
* Parse `.fhi` metadata correctly (ISO-8859-1 round-trip)
* Display submissions in a marking UI
* Enter marks (with the existing calculator) and tutor comments
* Persist crash-safely (CubDB) with encryption at rest
* Generate graded `.docx` for return to students
* Local authentication for a single tutor
* Single-binary distribution for Linux, macOS, Windows
* Audit log and basic DPIA-grade data protection

=== Explicitly out of scope for v1

These were oversold in the original docs. Each is weeks-to-months of work on its own. Re-evaluate post-pilot, based on whether a real user asks for it.

* All 15 marking-analytics stubs (Quality Bot, Bloom's alignment, argumentation analysis, syntactic complexity, divergence detection, etc.)
* Zotero integration
* Scheduling / Calendar / iCal export
* WASM plugin host
* Mobile (Tauri / Dioxus)
* Post-quantum crypto in user-facing flows (NIFs stay; not surfaced)
* Multi-tutor, multi-tenant, networked sync
* Course Management UI, Refinery (comment bank) UI
* AffineScript / Bebop / Zig frontends

== Phase 1 — Data plane (3–4 weeks)

The unblocker for everything visual. Sequence matters.

=== 1.1 `EtmaHandler.Repo` over CubDB (~1 week)

Schema:

* `{:submission, course, presentation, tma, oucu}` → submission record
* `{:audit, submission_id, ts}` → change event
* `{:snapshot, submission_id, ts}` → full state

Functions: `put/2`, `get/1`, `list/1`, `delete/1`, `snapshot/1`, `history/1`, `restore/2`.

Tests: property-based round-trip, concurrent writer stress, kill-and-restart durability.

=== 1.2 FHI parser/generator (~1 week)

* [ ] Acquire ≥5 real anonymised `.fhi` samples → `test/fixtures/fhi/`
* [ ] Implement `parse/1` with SweetXml; handle ISO-8859-1
* [ ] Implement `generate/1`; byte-identical round-trip on the fixtures
* [ ] Date parsing via Timex (FHI format: `23-Nov-2025 23:52:29`)
* [ ] Property tests: `parse(generate(x)) == x`

=== 1.3 Bouncer (~3–4 days)

* [ ] Fix the broken `init/1` (currently references undefined `download_dir` and `%__MODULE__{...}`)
* [ ] Wire `FileSystem` → filename regex → SweetXml parse → `Repo.put` → PubSub broadcast
* [ ] Quarantine path for invalid files
* [ ] Integration test: drop fixture into tmp dir, observe Repo write within 2s

=== 1.4 `.docx` generator (~1 week)

* [ ] Pick a library or implement a minimal docx writer (it's zipped XML)
* [ ] Insert tutor comments + scores into a template
* [ ] Test: output opens cleanly in LibreOffice and Word

**Exit criteria:** real `.fhi` can be dropped, ingested, marked via REPL, exported to a valid `.docx`.

== Phase 3 — Auth + data protection (3–4 weeks)

Non-negotiable: this handles student PII (OUCU, personal ID, address per FHI schema).

=== 3.1 Authentication (~1 week)

* [ ] Single-tutor passphrase login (Argon2id, already in deps)
* [ ] Session pipeline in router
* [ ] Idle lock
* [ ] WebAuthn integration if painless; defer to v1.1 if it eats >3 days

=== 3.2 Encryption at rest (~1 week)

* [ ] Finish stubbed `EtmaHandler.Crypto.encrypt/3`
* [ ] Wrap Repo writes with XChaCha20-Poly1305 (NIF works)
* [ ] Key derivation: Argon2id from passphrase; master key in OS keystore (libsecret / Keychain / Credential Manager), **not on disk**
* [ ] Test: kill app, grep CubDB files, confirm no plaintext PII

=== 3.3 DPIA + governance (~1–2 weeks; partly elapsed time)

* [ ] Data Protection Impact Assessment doc
* [ ] Retention policy aligned with OU rules
* [ ] GDPR data-subject endpoints: export-all / delete-all by OUCU
* [ ] Audit log of every Repo read/write, tied to authenticated tutor
* [ ] Rewrite `SECURITY.md` with a real threat model (replace generic boilerplate)

Start 3.3 in week 1 of this phase, in parallel with engineering — institutional review has its own clock.

**Exit criteria:** could be handed to your institution's DPO without embarrassment.

== Phase 4 — Release engineering + pilot (2–3 weeks)

=== 4.1 Burrito release (~1 week)

* [ ] Actually build a release; resolve every issue
* [ ] Smoke-test the binary on Linux x86_64, one macOS, Windows
* [ ] Non-technical install docs
* [ ] Fix `ApiController.health/2` — `Repo.get(:health_check)` is currently undefined

=== 4.2 CI sanity (~2–3 days)

14 workflows is too many. Collapse to one `elixir-ci.yml` doing deps → compile → test → dialyzer → credo → release-build. Park governance / scorecard / dogfood-gate workflows until they earn their keep.

=== 4.3 Pilot (~2 weeks calendar)

One real tutor (probably you) marks one real assignment cycle. Triage every bug; fix the showstoppers; iterate on UX.

**Exit criteria:** someone other than the author marks an assignment successfully.

== Top risks

. **FHI edge cases.** Get real samples in week 1 of Phase 1, not week 3. Undocumented variants are likely; phase 1.2 could double.
. **CubDB scale.** Fine for one tutor; do not assume it scales to a department. Re-evaluate if v2 needs multi-tutor.
. **WebAuthn rabbit hole.** Fall back to passphrase-only if it eats >3 days.
. **DPIA process.** Institutional, not technical. Start in week 1 of Phase 3.
. **Solo-dev burn.** The cut list is aggressive on purpose; resist re-adding scope.
